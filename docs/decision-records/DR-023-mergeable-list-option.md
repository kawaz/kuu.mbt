# DR-023: マージ可能リストオプション — `+/-` 修飾子と `...` 展開

type: decision

## 背景

業務ツールの CLI で「修飾子付きカンマ区切りリスト」パターン（`--fields "+extra,-excluded,..."`）を独自実装した経験から、このパターンが汎用的な需要を持つことが判明。実装は数百行、テストも同規模になり、CLI パーサ側で first-class サポートすることで、ユーザーが毎回同じマージロジックを自前実装する必要をなくす。

### 着想元の実装パターン（参考）

```bash
# フル上書き（修飾子なし）
--fields "method,url,status"

# マージモード（+/- 修飾子あり）
--fields "+extraField,-excludedField"

# 残りフィールド展開
--fields "timestamp,...,-session"
```

CLI パーサは `--fields` を単なる文字列として受け取るだけで、マージロジックは全てアプリケーション層で自前実装。この「あったら使うが自作はダルい」領域を kuu がカバーする。

## 提案する仕様

### 構文

カンマ区切りのリスト。各要素に `+`（追加）/ `-`（除外）修飾子を付与可能。`...` は「残りのデフォルト要素」を表す。

```
要素     := 修飾子? 識別子
修飾子   := '+' | '-'
識別子   := [^,]+
特殊要素 := '...'

リスト   := 要素 (',' 要素)*
```

### セマンティクス

#### 1. 平文リスト（修飾子なし） → フル上書き

```bash
--columns "name,age,email"
# → ["name", "age", "email"]（デフォルト/環境変数を完全に置き換え）
```

修飾子（`+/-`）も `...` も含まない場合、ベースを無視して指定値で全置換。「単純な指定ほど強い」という直感的な UX。

#### 2. `+` 修飾子 → 追加

```bash
# デフォルト: ["name", "age", "email"]
--columns "+phone,+address"
# → ["name", "age", "email", "phone", "address"]
```

ベースリストの末尾に追加。既にベースに存在する要素への `+` は no-op（重複排除）。

#### 3. `-` 修飾子 → 除外

```bash
# デフォルト: ["name", "age", "email"]
--columns "-age"
# → ["name", "email"]
```

ベースリストから該当要素を除去。

#### 4. `+` と `-` の混在

```bash
# デフォルト: ["name", "age", "email"]
--columns "-age,+phone"
# → ["name", "email", "phone"]
```

左から順に処理。同一要素への `-` → `+` は「削除してから追加」（末尾に移動する効果）。

#### 5. `...`（rest）展開

```bash
# デフォルト: ["name", "age", "email"]
--columns "id,...,-age"
# → ["id", "name", "email"]
```

`...` の位置にベースリストの未列挙要素を展開。`-` で除外された要素は展開対象外。`...` により要素の順序制御が可能になる。

#### 6. 意図の自動判定

| 入力パターン | 判定 | 動作 |
|---|---|---|
| 平文のみ (`"a,b,c"`) | フル上書き | ベースを無視 |
| `+/-` を含む (`"+a,-b"`) | マージ | ベースに対する差分適用 |
| `...` を含む (`"a,...,-b"`) | マージ + 順序制御 | `...` 位置にベース展開 |

この自動判定がユーザー UX の肝。明示的な `--merge` / `--override` フラグを別途用意する必要がない。

### 3段マージ（環境変数連携）

```
デフォルト値 → 環境変数 → CLI引数
```

Step 5（defaults マルチソース）の環境変数連携と組み合わせることで、3段階の優先度マージが実現する:

```bash
# 1. コード上のデフォルト
mergeable_list(name="columns", default=["name", "age", "email"])

# 2. 環境変数で組織共通設定
MYAPP_COLUMNS="+department,-email"
# → ["name", "age", "department"]

# 3. CLI引数でユーザー個別調整
--columns "-age,+phone"
# → ["name", "department", "phone"]
```

各段階で同じマージセマンティクスが適用される。

## 既存アーキテクチャとの統合

### Accumulator ベースの実装

現行の `Accumulator[T, U]` パターン（`append_string` 等で使用）の拡張として自然に位置づけられる:

```
Sugar層:    mergeable_list(name="columns", default=[...])
               ↓
Convention層: register_option(node) + マージ用 Accumulator
               ↓
Pattern層:   make_or_node([...])
               ↓
Core層:      ExactNode + 消費ループ
```

- **値の保持**: `Ref[Array[String]]` にベースリストを保持。`+/-/...` の解釈は reducer 内で処理
- **FilterChain との連携**: マージ後の配列に対して FilterChain でバリデーション可能（`each(one_of(allowed_values))` 等）
- **post_hooks**: マージ演算は reducer 内で完結。post_hooks は最終バリデーション用

### コンビネータ API（概念）

```moonbit
let columns : Opt[Array[String]] = p.mergeable_list(
  name="columns",
  default=Initial::Val(["name", "age", "email"]),
  // オプション: 許可リスト制約
  // post=Filter::validate(fn(list) { list.iter().all(fn(x) { allowed.contains(x) }) }),
)
```

## スコープ

### kuu に含めるもの

| 機能 | 理由 |
|---|---|
| `+/-` 修飾子によるマージ | 汎用性が高い。設定ファイル、フィルタ、カラム選択等 |
| `...`（rest）展開 | 「残りはデフォルト通り」の表現。全項目を知らなくても使える |
| 環境変数との階層マージ | Step 5 との統合。3段マージが自然に実現 |
| 平文/修飾子の意図自動判定 | UX の核。毎回自前実装すると仕様ブレが起きる |
| 重複排除 | マージ操作の基本保証 |

### kuu に含めないもの

| 機能 | 理由 |
|---|---|
| スライス表記 (`field[0]`, `field[:10]`) | ドメイン固有。FilterChain で十分対応可能 |
| フィールドの型解釈 | アプリケーション層の責務 |
| ネストしたオブジェクトのマージ | 複雑すぎる。YAML/TOML パーサの領域 |

## 差別化要因

28パーサの調査（2026-02-24-cli-parser-mega-survey.md）において、`+/-` 修飾子付きリストマージを first-class でサポートするパーサは確認されていない。多くのパーサは `append`（累積追加）を提供するが、「ベースに対する差分指定」「フル上書きとマージの意図自動判定」を組み込みで提供するものはない。

## ユースケース例

| CLI | オプション | 用途 |
|---|---|---|
| ログビューア | `--fields` | 表示カラムの選択・並び替え |
| テストランナー | `--suites` | 実行スイートの選択・除外 |
| デプロイツール | `--targets` | デプロイ先の選択・除外 |
| フォーマッタ | `--rules` | 適用ルールの追加・除外 |
| API クライアント | `--headers` | リクエストヘッダの追加・除外 |

いずれも「デフォルトがあり、ユーザーが部分的に調整したい」パターン。

## 実装順序

1. Step 5（環境変数連携）の実装が前提
2. マージセマンティクスのパース関数（`+/-/...` の解釈）
3. `mergeable_list` コンビネータの Sugar 層実装
4. 3段マージの統合
5. ヘルプ表示への反映（デフォルト値の表示方法）

## 未決事項

- **コンビネータ名**: `mergeable_list` は仮称。`list_opt`、`set_opt` 等も候補
- **セパレータのカスタマイズ**: カンマ以外（スペース、コロン等）を許容するか
- **`...` の複数出現**: `"a,...,b,...,c"` を許容するか（現時点では単一のみを想定）
- **大文字小文字の扱い**: マージ時の一致判定で case-insensitive を選択可能にするか
- **ヘルプ表示**: `+/-` 構文の説明をどこに表示するか（オプション説明文 / フッタ / 専用セクション）
