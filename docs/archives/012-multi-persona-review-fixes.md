# DR-012: マルチペルソナレビューによる設計修正

## 経緯

ユーザーの指示で「先食い最長一致 + resolve廃止」の新設計に対して4ペルソナで並列レビューを実施。

### レビューペルソナ

1. **パーサ理論**: 形式言語・構文解析の観点
2. **API設計/DX**: ユーザー体験・API一貫性の観点
3. **MoonBit実装者**: MoonBit言語の制約・実装可能性の観点
4. **エッジケース破壊者**: 境界条件・破壊的テストの観点

### 発見された致命的問題

#### 1. try_reduce の raise ParseError と投機実行の矛盾（全4ペルソナ）

消費ループは全候補に投機的に try_reduce を呼ぶ。raise ParseError があると:
- 非採用候補のエラーで全体が落ちる
- 試行順序が結果を左右する

**解決**: try_reduce から raise を除去。内部で try? → catch → Reject。

#### 2. 完全一致 vs プレフィックスマッチの優先度未定義（ペルソナ1, 4）

`--output` が完全一致、`--output-format` がプレフィックスマッチ → consumed 同数で曖昧エラーになるが、完全一致が優先されるべき。

**解決**: MatchQuality (Exact/Prefix) を追加。消費ループに完全一致フィルタステップを追加。

#### 3. `--` の force_positional シグナル欠如（codex, ペルソナ2, 4）

`--` を通常 Opt として表現するが、消費ループへのシグナルが未定義。

**解決**: 消費ループ側で条件分岐。name="" + greedy + global の Flag が Accept → force_positional 遷移。

### 追加の修正

- **ResultMap + Opt.slots の二重記述**: Parser.refs + getter に一本化
- **P モード短縮オプション展開**: フローを明記
- **ReduceCtx.get のセマンティクス**: 投機実行中は stale read であることを明記

## 決定事項

1. **try_reduce は raise しない** — 投機実行の安全性を保証
2. **MatchQuality で完全一致を優先** — 消費ループに完全一致フィルタを追加
3. **force_positional は消費ループ側の状態変数** — TryReduceResult にシグナルを追加しない
4. **Opt[T] は getter 方式に統一** — slots フィールドは廃止
5. **P モードでも短縮オプション展開を適用**
6. **ctx.get() は stale read** — 投機実行中は前 iteration の確定値のみ

## 不採用案

| 案 | 不採用理由 |
|---|---|
| try_reduce で raise を許可しつつ投機実行で catch | 各ノードの try_reduce が副作用なしである保証が必要だが、raise を許可すると呼び出し側の複雑さが増す。try_reduce 自体で catch する方がシンプル |
| TryReduceResult に force_positional シグナル追加 | 特殊ケース（`--` のみ）のために汎用型を汚すのは不適切。消費ループ側の条件分岐で十分 |
| MatchQuality を consumed の重み付けで表現 | consumed が同じだが quality が異なるケースを自然に処理できない。独立フィルタの方が明確 |
| Opt[T].slots を残して getter と共存 | 二重記述による混乱。getter に一本化する方が一貫性がある |

## 付録: 生のチャットやりとり

### ペルソナレビューの開始

> **User**: さてまた構造が変化したわけだが、新しい先食いリデューサーの変更によって、既存だと1ステップずつだった引数消費が何かの枷になってて歪んでいた設計とか、今回の変更によって新たな新しいアイデアが無いかとか、既存処理を更に良くできる可能性がないか、または論理的的な矛盾が無いかなど、複数のペルソナで検討してみて下さい。

### 各ペルソナの主要な指摘

**ペルソナ1（パーサ理論）:**
- 致命的: try_reduce の raise と投機実行の矛盾

**ペルソナ2（API設計/DX）:**
- ResultMap+slots を Parser.refs+getter に一本化すべき

**ペルソナ3（MoonBit実装）:**
- 致命的箇所なし
- `fn(...) raise ParseError { ... }` クロージャ構文で対応可能

**ペルソナ4（エッジケース）:**
- 致命的: raise+投機実行矛盾
- 致命的: 完全一致 vs プレフィックス優先度未定義
