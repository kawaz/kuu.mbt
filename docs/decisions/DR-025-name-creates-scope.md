# DR-025: name が結果スコープを作る、露出は最も浅い name 層

## 決定

### name が結果スコープを作る

- **name を持つノード = 結果スコープ (結果オブジェクトのキー) を作る。**
- **name 無しノード = 透過。スコープもキーも作らず、値の畳み方だけ効かせて値を親に流す。**

children の有無ではなく **name の有無**でスコープが決まる。

### 露出規則 (最も浅い name 層)

結果への露出は以下の規則で決まる:

1. 根から降りていって、最も浅い (祖先側の) name 層で止める。
2. その層にある name 持ちノードを **全て** 結果キーにする (同じ層の name 兄弟は全部拾う)。
3. それより深い name は、止めた層のノードが作る **子スコープ** に属する (再帰適用)。

「最初の name アリ要素」= 根からの巡回で最も浅い (祖先側) の意味。同じ深さの兄弟 (例: r/g/b) は全て拾われる。「最初の1つだけ拾って兄弟を捨てる」ではない。

### object は独立構造でなく露出の帰結

kv (object 的な結果) は専用構造を持たない。**name を持つ子が並べば結果が自然に kv になる**。`{r,g,b}` は `[{name:"r",...},{name:"g",...},{name:"b",...}]` (seq に name 持ち子) の露出結果。

- name 無しの子は結果に出ない (exact は値なしなので当然、値プリミティブも name なしなら値は捨てられる)。
- name 持ちと name 無しの混在は問題にならない (name 無しは消えるだけ、kv には name 持ちだけ現れる)。
- kv を作りたい単位には name を付けるのが自然な書き方。「name 無しで値を持つ子を kv に入れる」異常ケースを AST が救済する必要はない。

## --color の4パターン (露出規則の実証)

| | or に name | 子に name | RGB 畳み方 | 結果 |
|---|---|---|---|---|
| A | ✓ color | ✗ | seq → 配列 | `color: [n,n,n] \| string` |
| B | ✗ | ✓ 枝ごと | seq → 配列 | `colorRGB? \| colorName?` (排他) |
| C | ✓ color | ✓ 枝ごと | seq → 配列 | `color: {rgb} \| {name}` (Partial union) |
| D | ✓ color | RGB内 r/g/b | name 持ち子 → kv | `color: {r,g,b} \| string` |

A/B の差は「or に name を付けるか、子に name を付けるか」だけ。C は二重 name で、or の排他性により「両キーを持つ object」ではなく Partial union ({rgb} か {name} の一方) になる。C は型注釈としては正しく、許す (DR-021 の「曖昧でなければ許す」)。name があればそこにスコープ階層ができ、そのスコープ内の各枝の name 持ち要素を露出する。

## 既存記述の訂正

元の設計にはスコープ生成基準が2箇所あり食い違っていた:

- DR-002 の表: 「スコープ範囲 = name を持つ / 素通し」← **name 説**
- AST-SPEC §11.1: 「children を持つ要素は自動でスコープを作る」← **children 説**

A〜D の検討で **name 説が正しい**と実証 (D の name 無し object は children を持つがキーを作らず、中の r/g/b が親に出る。children 説では D が表現できない)。AST-SPEC §11.1 の「children でスコープ」は訂正対象。

ただし children も無関係ではなく役割が違う:

- **結果スコープ** (露出) = name が作る ← 本 DR
- **lexical スコープ** (ref/link/inherit の解決範囲) = children が作る ← 別概念、未詰め (保留)

2つを「スコープ」の一語で混同していたのが前回の散らかりの一因。用語を分ける。lexical スコープ自体の議論は今回は棚上げ。
