# DR-020: command の復帰・途中分岐は専用概念を持たず構造プリミティブで組む

## 決定

command の「スコープ復帰」「途中分岐」「再帰」といった消費パターンに **専用概念 (フラグ・専用 type) を AST に持たせない**。これらは構造プリミティブ (exact / or / serial / multiple) の **組み合わせでユーザが組む**。

例 (kawaz):

```
type: repeat(= multiple), children: [ --command, (or: サブコマンドs) ]
```

のように、既存プリミティブを構成すれば「同じ command を何度も」「途中で別 command に分岐」「消費後に親へ復帰」が表現できる。

**曖昧さなく最長一致で解決できる構造を組むのはユーザの責務**。組めなければそれはユーザの設計の問題であり、AST は少数のプリミティブを提供するだけ。

## 経緯

Claude は「復帰型 (有限消費して親に戻る) / 非復帰型 (残り全部を食う)」を AST のフィールドで区別すべきか、配置で決めるべきか、と問うた。kawaz:

> 好きにすれば良いと思うが。type:repeat, children(--command, (or:サブコマンドs)) みたいにするとか。面倒になる部分はユーザが工夫してパース可能にする責務を負う的な。

これは CONTEXT.md の反省パターン1 (概念を勝手に膨らませる) を Claude が再び踏みかけたのを、kawaz が「専用概念は要らない、プリミティブの構成で十分」と引き戻したもの。

`docker run IMAGE CMD...` のような途中分岐も、入れ子 command の復帰も:

- positional を消費 → その後 command 分岐
- command スコープに入り有限消費後に親へ復帰、または特殊セパレータ (`--` 等) で明示復帰

どの形でも、**最長一致で一意に解決できる限り定義として許す**。曖昧になった時だけ実行時 ambiguous (DR-021)。

## 効果

- 「復帰」「再帰」「途中分岐」が AST 語彙から消える (DR-002 の同型・DR-015 の伝搬と地続き)。
- AST が提供するのは少数の構造プリミティブと値伝搬規則だけ。
- 複雑な消費パターンの表現可能性とパース可能性の保証は、ユーザの構造設計に委ねられる。
