# DR-021: 最長一致パースと ambiguous、露出キーの一意性検査は実行時

## 決定

### パースの成功条件

> **曖昧さなく最長一致で全ての引数を解決できたら成功。**

途中で曖昧さが発生した時点で **ambiguous エラー**。複雑な競合解決ルールは持たず、最長一致で一意に解ければ成功、解けなければ ambiguous。

### 露出キーの一意性検査は実行時

name 無しリスト/構造の子 name が親スコープに **露出 (expose)** するケース (例: `--color R G B` の R/G/B が親に露出) において、name の一意性検査は **パース実行時に発動**する。

- 最長一致で解決できる限り、定義上キーが衝突しうる構造でも **許す**。
- 実際に同一入力で両方が露出して衝突した時のみ ambiguous エラー。

理由 (kawaz):

> 定義上ありえても実用時点では同時に使われることはない筈と考える事情があるだろうと期待できる。

利用者が「実用上は同時に使われない」と知っているケースを、定義エラーで殺さず実行時 ambiguous に委ねる。利用者の知識を信頼する設計。

### 静的バリデータは warn のみ

潜在的に ambiguous になりうる定義を **静的に検出して warn できるべき**だが、**reject はしない**。

- 「上限なし multiple が positional 列に複数」→ potential ambiguous を warn (DR-019)
- 露出キーが衝突しうる構造 → warn
- warn はする、reject はしない、の二段構え。

## 露出キーの型と union

同一スコープに露出する複数のキーが衝突しうる場合、型の扱いは状況で分かれる:

### (a) 同一入力で両方が値を持ちうる (真の衝突)

ambiguous の対象。最長一致で片方しか選ばれないなら衝突なし。両方選ばれうるなら ambiguous エラー。**型は無関係** (両立しないので同型である必要なし)。

### (b) 排他的にどちらか一方しか現れない (or 的露出)

同じキーに異なる枝から異なる型が入るケース (例: `--color R G B` (number×3) と `--color none` (string))。

- **同じ型なら** そのまま露出 (union が縮退して単一型)。
- **違う型なら**: そのまま許す → 結果の静的型が union になる。union を嫌うなら `export_key` で別名を付けてキー空間を分離。

**「同じ型であるべき」を強制ルールにはしない。** 強制すると「実用上ありえない衝突」のために正当な定義まで弾き、上記の精神に反する。指針として「型が一致しなければ union、嫌うなら export_key」に留める。

### スコープ階層が違えば対象外

(a)(b) いずれも **同一スコープに露出したときだけ**発生。階層が分かれていれば検査対象外。この除外条件は明示する。

## 経緯

kawaz の整理:

> パースのロジック自体は kuu で構想した曖昧さなく最長一致で全ての引数を解決出来たら成功、のロジックで良い。その場合、途中で曖昧さが発生した時点で ambiguous エラーになる。最初からそのリスクがあるか無いかのバリデーション自体はできるべきだが、それでも良いと考えて利用する分には実行時エラーに回す選択肢自体はあって良い。

型については:

> 型システム的にスコープに露出する可能性のあるキーは同じ型であるべきかもしれない? そうでないなら別名を付けるべきくらいかな。あとはスコープ階層が違うならそもそも問題無いし。

→ 強制せず指針に。

## 効果

- name 無しリストの子 name 露出 (前回 CONTEXT.md 論点 B/C) が実行時に自然回収され、定義時に潰しきる必要がなくなる。
- 「明示性重視」と「利用者の知識を信頼」が両立 (warn はする、reject はしない)。
- 露出キーの型衝突が union / export_key で扱われ、強制ルールを増やさない。

## 注

本 DR はパースの「成功条件」と「検査タイミング」を定義するもので、最長一致の具体的アルゴリズム (実装ロジック) には踏み込まない。AST が表現する構造が「一意に解けるか」を保証する形になっているか、までが対象。
