# DR-017: command は定義時1級、パースループ上は同型

## 決定

`command` は **定義時 (UsefulAST) には1級の概念**として扱う。`option` / `positional` と並ぶ第一級の構造語彙であり、専用フィールド (`commands`) や `type: "command"` で明示的に書ける。

ただし **パースループにかける際 (AtomicAST) には同型要素**として扱う。command は構造プリミティブ (exact + or + seq 等) の組み合わせへ展開され、パーサは「name でトリガしうる children を持つ要素」という同型表現の上で動く。

つまり:

```
定義時 (UsefulAST):  command は1級     ← 書きやすさ・読みやすさ優先
    ↓ 展開
パース時 (AtomicAST): 同型要素に分解    ← パースループの単純さ優先
```

## 経緯

DR-002 で「すべての要素は同型、CLI 慣習名はシュガーの愛称」と整理した。しかしその勢いで **定義語彙まで同型に潰してしまった** (DR-004 で `commands` 専用フィールドを却下し、positionals 内 or のみに) のが行き過ぎだった。

kawaz の方針転換:

> 内部実装はまた再度詰め直すが command という概念は定義時には1級と考えるべきと思い直したのが今回の根っこ。ただし内部でパースループにかける際は同型要素として扱われる想定。

トップダウンで設計し直す中で、「定義時に command を1級に見せる」ことと「パース時に同型化する」ことは**矛盾しない**と確認された。これは DR-001 の2層 AST 構造そのものであり、UsefulAST に command を1級で残し、AtomicAST で同型に落とす、という分離で両立する。

## 効果

- DR-002 の同型の発見 (パース表現) を保ちつつ、DR-004 で却下した command 1級表記が復活できる (DR-018)。
- 「定義は1級、パースループは同型」が type の省略/復元で繋がる (DR-019)。
- DR-002 の「同型」は AtomicAST の話、と適用範囲が明確化される。

## 関連

- DR-002 (全要素同型) — 適用範囲を AtomicAST に限定
- DR-004 (options/positionals 分割) — commands 却下を DR-018 で部分的に撤回
