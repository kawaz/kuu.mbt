# DR-037: filter の Reject/Error 区別、解けた枝の数による結末分類

## 決定

### filter は Reject と Error を区別する

filter (および mapper) のレスポンスは2種類を区別する:

- **Reject**: 「この枝ではない、他を試して」→ エラー保持せず、その枝を選択候補から外す
- **Error**: 「この枝のつもりだが値が不正、ユーザに伝えたい」→ エラーを保持

kuu core の TryResult (Accept/Reject/Error) と並行する。filter レベルでも同じ区別を持ち込む。kuu core の現状は filter は ParseError 一種類だけだが、本 AST 設計として2種類を区別する。

### 「解けた枝の数」で結末が決まる

or のような選択構造で複数の枝を試した後の結末は、**「解けた (Accept 完了した) 枝の数」**で決まる:

| 解けた枝の数 | 結末 |
|---|---|
| 0個 | 全体失敗 (保持された Error を見せる、複数あればその扱いは下記) |
| 1個 | その枝で確定 (他枝の Reject/Error は捨てる) |
| 2個以上 | ambiguous エラー (どれも全引数消費に成功) |

「解けた」= filter Error 含めて全段が通った枝。filter で Reject/Error が出た枝は「解けなかった」枝として脱落するが:

- Reject なら エラーは保持しない (静かに脱落)
- Error なら エラーを保持 (全体失敗時の表示候補に)

これにより「他ルートが通るならエラーは捨てられる」が成立する (DR-021 「曖昧でなければ許す」原則の自然な帰結)。

### 「最長一致」は枝内の競合にのみ効く

DR-021 の「最長一致で一意に解ければ成功」は、**枝内の消費長競合** (例: `--color R G B` (3引数) と `--color R G` (2引数) の選択) には効く。

ただし **option と positional の境界をまたぐ経路の長さ**が同じ場合は、最長一致が意味を持たず ambiguous:

```
入力: --color 255 0 0 ("0" が実在ファイル、positional 受け先あり)

枝A: --color R G B (option で3消費) + positional 0個
枝B: --color <name> (option で1消費) + positional 2個 ("0", "0")

両方とも全引数消費 → 経路レベルで曖昧 → ambiguous
```

これは「2通りの成功」であり、DR-021 の ambiguous エラーの典型。

### filter Error が複数発生した場合の表示

複数の枝が Error で失敗した場合、ユーザに見せるエラーは複数の候補から1つ選ぶ:

> 雑に答えてたが深い方かな。(kawaz)

→ 一番奥まで進んだエラー (構文的にユーザの意図に近い) を見せる方針。具体的な選択ルール (深さの測り方、tie-breaking) は実装側で詰める論点として残す。

ただし複数のエラーを両方ダンプする選択肢も妥当 (kuu の明示性重視と整合)。これは出力モードの問題で、AST 設計上は両方の情報を保持する。

## DR-021 との関係

本 DR は DR-021 の補完。DR-021 が確定したのは:

- 曖昧さなく最長一致で全引数解決できれば成功
- 曖昧なら ambiguous エラー
- 露出キーの一意性検査は実行時、warn はする reject はしない

本 DR が追加するのは:

- filter レベルの Reject/Error 区別
- 「解けた枝の数」による分類 (0/1/2+)
- 「他ルートが通るならエラーは捨てる」の正確な定式化

## 経緯

### 「途中エラーで他ルートが通れば捨てる」の確認

kawaz:
> 途中エラーは評価を遅延して全ルートチェックして他が通るならエラーは捨てられるみたいになってませんでしたっけ?

→ DR-021 の自然な帰結だが明文化されていなかった。本 DR で追記。

### filter Reject/Error の必要性

`--color 256 0 0` のようなケースで、filter (in_range) が「Reject (他の枝を試して)」を返すか「Error (この枝だがダメ)」を返すかで結末が変わる:

- Reject なら → 他の枝へ
- Error なら → エラー保持、全体失敗時に表示

ユースケースとして両方ありえる。AST 設計として区別を入れる。

### 「最長一致が機能しない場合」

> まぁ最初から明言してるよね。「一意に」解けなければ、と。(kawaz)

DR-021 の「一意に解ければ」を厳密に読めば最長一致は「一意化の手段の一つ」であり、それで一意化できない経路レベルの曖昧性は ambiguous になる、というのは DR-021 の射程内。本 DR で「経路レベルの曖昧性も ambiguous」を明示する。
