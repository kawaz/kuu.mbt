# DR-029: link の見直し — 値同期、1実体:N参照、固定パス DSL、遅延解決

## 決定

### link は値同期 (ref/type と直交)

link は「別経路・別パースの入口を、同一の結果セルに合流させる」機構。ref/type (定義の継承) とは直交する。

| | 定義 (パース方法) | 結果セル |
|---|---|---|
| ref/type (継承) | 共有 | 別々 |
| link (値同期) | 別々 | 共有 |

例 (kawaz): カウンタ short `-v` と `--log-level N` が同じ logLevel セルを共有する。

- `-vvv` → counter として increment、値 3
- `--log-level 3` → number として 1引数消費、値 3
- 結果は logLevel: 3 が1個だけ

入口の綴りもパース方法 (counter vs number) も違うが、同じセルに合流する。これは ref/type では書けない (パース方法が別なので type 共有不可)。link の唯一無二の役割。

### 値の合流は「操作の時系列適用」

同一セルに複数経路が書いたら、各入口が自分の操作をセルに出現順で適用する (後勝ちではなく操作列):

`-vv --log-level 5 -v` → 0 →(increment×2) 2 →(override) 5 →(increment) 6

counter は increment 操作、number は override 操作。accumulator のセマンティクスがセルに紐づく。

### 1実体 : N参照

link を書いた側が参照、link されていない name 持ちが実体。

> 実体一つに対してリンク3個で4オプションが実体一つを共有してると考えたら自然 (kawaz)

実体をどちらの入口にするかは設計者の自由だが、基本は「link を書いた方が参照、書かれていない方が実体」。定義の自然な読みは「link 先 = 実体」。

実体側に特別な印 (「ここは link されうる」マーク) は不要。name で名乗っていれば参照側が link:"name" で辿れる。

### link 先は固定パス DSL

```
path := name ('.' name | '[' int ']')*
```

- `.name` : フィールドアクセス
- `[int]` : 固定インデックス。非負 (先頭から) または負 (末尾から、-1 = 最後)
- 動的キー `[var]`、ワイルドカード、スライス `[0:2]`、再帰 `**` は無し

例: `link:"logLevel"`, `link:"timerange.since"`, `link:"color.rgb[0]"`, `link:"color.rgb[-1]"`

### 解決は遅延 (実行時)

静的解決はしない。理由: 型の内部構造を AST が知らないケースがある。

> datetime は型の構造を知らないので link の解決は実行時にしかできない (kawaz)

`type:"datetime"` の内部構造は value_parser (AST 外の関数) が実行時に作るオブジェクトにしかない。AST ロード時には解決しようがない。よって遅延解決が必然。

### 解決失敗 = パース失敗

link 先が解決できなければ、エラーを特別扱いせず「その経路のパース失敗」として扱う (DR-021)。

> そこでエラーになるならパース可能と解釈されずパース失敗になるだけ (kawaz)

他に成立する解釈があればそれが選ばれ、無ければ全体がパース失敗。or の枝をまたぐパスも静的に禁止しない (実行時に枝が確定し、link 先があれば解決、無ければ経路が落ちるだけ)。

### セレクタ言語 vs 固定パス

- 却下: 動的・汎用セレクタ言語 (リフレクションベースの汎用クエリ)
- 採用: 固定パス (`.name`/`[int]`/`[-1]`) の実行時解決

線引きは「セレクタ言語 (動的・汎用)」却下、「固定パス (静的記述・実行時に辿るだけ)」採用。固定パスを実行時に辿る最小機構なら各言語で書ける。
