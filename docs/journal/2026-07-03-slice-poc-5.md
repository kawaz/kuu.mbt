## Task 1: 効果記述子 (DR-045) — phase13, 7 test

値セル操作を純データ `{op, operand?}` で表現。全 binding に `op` を載せ、効果列を一様化した。

### 実測 (契約割れなし)

- **committed の派生**: `committed(Set/Default/Empty)=true`、`committed(Unset)=false` を直接固定。
  別フィールドでなく op の純関数、という DR-045 §2 の主張を実装で確定。
- **op が同一性成分 (DR-038 精密化)**: 同一セル `x` に同一値 `false` を書く 2 エントリで、
  片方 `Set`・片方 `Unset` にすると **ambiguous:2** (効果が違う)。両方 `Set` なら 1 本合流。
  op を binding に載せた瞬間、dedup (`Array[Binding]` の構造比較) が op を含む効果列同一性に
  なった — 第 3 弾で無自覚に「エントリ情報を binding に残さない」設計にしていたのが、DR-045 の
  「op/operand を含む形に精密化」にそのまま乗った。
- **unset がラダーを開く / default がロックする**: color に env 席 (COLOR) + default "auto"。
  `--no-color` (unset) → 未 committed → ラダー後段の env が勝つ (`always@env`)。
  `--color red` (set) → committed → `red@cli`。`--color-default` (default) → committed だが
  値は default、**env を無視してロック** (`auto@default`)。引数なし → env (`always@env`、
  unset と同じ = 「触っていない」)。unset と default の唯一の差 (committed) がラダー挙動を
  分ける、を end-to-end で実測。
- **empty がコレクションを空にする**: `--tag a --tag b --clear-tags` (empty) → `{tag:[]}`。
  `--tag a --clear-tags --tag c` → `{tag:[c]}` (clear 後に再蓄積)。result builder の accumulate
  fold を op 順で処理し、Empty で list をリセットする形に落ちた。
- **variant lowering `{exact, link, effect}`**: `--no-ssl` (ssl の unset variant) が
  `Seq([Exact("--no-ssl"), EffMark("ssl", Unset)])` として greedy 面に落ちる (link は name 経由)。
  set variant (`--gzip` → compress="gzip") は Bind (set 縮退形) に落ちる。

### 契約の所見 / flag (メインへ)

**LOWERING §B.1 は variant の効果 lowering 形を示していない**。DR-045 §由来 が自ら
「variant の非 set effect の lowering が未定義という gap (LOWERING カタログ G-4/G-5)」と
明記している通り。PoC はこの lowering を `{exact, link (by name), effect (op)}` で確立した。
DR-045 §1 の例 (`{"exact":"--no-color","link":"color","effect":{"op":"unset"}}`) と一致。
**LOWERING §B.1 に variant 効果形 (B.1 の long installer 節への追記) を足すべき** — 新たな
矛盾ではなく DR が予告済みの gap を埋めた形。

## Task 4: `optional: true` = `repeat: {min:0, max:1}` 糖衣 (DR-043) — phase14 の 2 test

`ElemDef.optional` を追加、`elem_repeat(e)` で `{min:0, max:Some(1)}` に還元 (独立概念でない)。

### 実測

- **糖衣が閉包に同一還元**: `optional=true` の def と `repeat=rep(0,max=1)` の def が
  **canon_scope 完全一致** + parse も同一。optional は並行パスでなく 1 つの還元、を構造等価で証明。
- **completeness**: optional を使うが RepeatInst を外すと `unknown special vocab 'repeat'` エラー。
  optional は repeat installer の所有語彙 (独立概念でない) を実測で裏付け。

### 契約の所見 / flag (メインへ) — max=1 の結果形

**PoC は「accumulate (list) するのは閉包が ≥2 値を束縛しうる時 (unbounded / max≥2)。max=1
(optional 含む) は scalar セル (0-or-1)」という規則を採った** (`repeat_accumulates`)。理由:
flatten は MULTI cons を畳む機構で、max=1 なら値は高々 1 個なので scalar-or-absent が自然、
かつ optional の直観に一致。

**ただし DR-043 も DR-044 も max=1 の結果形 (list vs scalar) を明記していない**。DR-044 §1 の
「配列が既定」を字義通り取れば optional も 0/1 の配列になりうる。PoC は「≥2 の可能性で list、
max=1 で scalar」という境界規則を**設計判断として持ち込んだ**。DR に明記の余地あり (=
optional/max=1 を scalar とするか 0/1 配列とするか)。

## Task 5: min>=2 / max 有限の unfold (LOWERING §B.8) — phase14 の 5 test

`RepeatSpec.max` を追加。unbounded は既存 Ref 再帰、有限 max は `BoundedTail(head, budget, lazy)`。

### 設計判断: BoundedTail を porous 専用ノードに

有限 max を単純な入れ子 Opt (`head + Opt(head + Opt(head))`) で lowering すると、Or 枝が
scope_consume の `other =>` 経由で eval され **先食い (suppressed) を経由しない** + **取り分選好
(greedy/lazy) が効かない** (全 split を枝列挙 = ambiguous 化)。これを避けるため、有限 max は
`BoundedTail(head, budget, lazy)` を spine に置き、`bounded_tail_step` が budget を再帰で減らし
ながら stop/more を porous に (consume_head 経由 = 先食い保持、has_full オラクル = 取り分選好保持)
処理する。budget は spine 再訪でなく再帰深さで減るので固定 spine と両立する。

off-spine (eval が BoundedTail に到達する稀路) 用に `bounded_opt` で入れ子 Opt 等価形も用意。

### 実測 (契約割れなし)

- **min:2 (unbounded)**: `a` fail (min 未達)、`a b` → `[a,b]`、`a b c` → `[a,b,c]`。
- **max 有限 {min:1,max:3}**: `a` → `[a]`、`a b c` → `[a,b,c]`、`a b c d` → fail (4 個目が max 超過
  で消費不能・後続なし)。lowering は `head + BoundedTail(head, 2)` (Ref 再帰なし)。
- **exact {min:2,max:2}**: `a` fail、`a b` → `[a,b]`、`a b c` fail。
- **max × 取り分選好 greedy**: `xs{1,3} ys{1,3}` に `a b c d` → `{xs:[a,b,c],ys:[d]}` (左が最長取り分)。
- **max × 取り分選好 lazy**: xs を lazy に → `{xs:[a],ys:[b,c,d]}` (最短取り分)。
  → **bounded 尾部が Ref path と同じ greedy/lazy 選好を持つ**ことを実測 (取り分選好併用)。

## Task 6: 入れ子 repeat の下流バックトラック (取り分完成オラクルのスコープ相対化) — phase16, 1 test

DR-043 末尾が「親の消費が続くスコープ内の repeat では全体長基準の完成判定が壊れる」と明記した
論点を stress。

### 実測 — **DR 予告の限界を具体再現 (contract flag)**

command `seq` の body に greedy repeat `xs`、root に末尾 positional `tail` を置く。
`seq a b c` の正しい解釈は `{seq:{xs:[a,b]}, tail:c}` が存在する。しかし **取り分選好の完成
オラクル `has_full` が大域長 (4) を基準にする**ため、seq 内の greedy `xs` は `[a,b,c]` (大域末端に
到達) にコミットし、root の tail を試す前に `[a,b]` split を**枝刈り**する → tail に残る token が
なく **parse 全体が fail** (`fail:no complete parse`)。

これは DR-043 末尾 (「実装が枝刈りのために局所的な完成判定を使う場合、スコープの消費境界に相対化
しないと近似が壊れる ... 親の消費が続くスコープ内の repeat では不可」) の**具体再現**。第 3 弾の
has_full flag と同根。**新たな矛盾ではなく DR 自身が予告した限界**を concrete case で固定した。
根治 (完成オラクルのスコープ境界相対化 / CPS 化) はフルエンジンの設計論点として持ち越し
(第 3 弾でも「最優先の設計論点」と記録済み)。symptom-fix (テスト側で誤魔化す) はせず、破れる
事実を test として可視化した。

## Task 1: 一様配列への修正 (DR-044 §1 裁定) — phase14 更新, +1 test

kawaz 裁定: repeat / multiple / separator を宣言した要素は max に依らず**常に配列**。`optional: true`
(= `repeat:{min:0,max:1}` 糖衣) も `[]` / `[x]`。scalar が欲しければ collector (unwrap_single) で畳む。

### 実装

- `ensure_entity` の max ゲート (`repeat_accumulates`) を撤廃。repeat は max に依らず accum "flatten"。
  `repeat_accumulates` 関数は削除 (unused)。
- **空 accumulate セルの `[]` 表示** (一様配列の肝): 発火 0 回の accumulate セルは、第 5 弾までは結果に
  現れなかった (同名 binding が無い → キー欠落)。裁定の `[]` を満たすため result builder を **スコープ
  限定の accum セル注入**に対応させた: `accum_names` (フラット名) → `accum_cells` (result-path 付き)、
  `build_result` に current path を通し、その path に home を持つ accum セルが named に無ければ `[]` を注入。
  path 付与により、command scope 内のセルを top level に誤注入する事故を防ぐ (`serve` 未起動なら
  `serve` scope 自体が結果に現れないので注入も走らない)。
- **unwrap_single collector** (DR-034 縮退, opt-in scalar): 長さ 1 配列を要素へ再帰的に unwrap。
  `[x]` -> `x`、`[]` は unwrap 対象なし -> `[]` のまま。from_entries と同じ RVal->RVal collector 形。

### 実測

- `entity_accum(optional)` が "scalar" -> **"flatten"**。`optional []` -> `{x:[]}`、`optional [a]` -> `{x:[a]}`
  (第 5 弾は `{}` / `{x:a}`)。テストの意図コメントも新仕様 (一様配列) に書き換えた (テスト改変でなく仕様変更)。
- optional と `repeat:{min:0,max:1}` は canon_scope 完全一致を維持 (糖衣還元は不変)。
- `unwrap_single([a])` -> `a`、`unwrap_single([])` -> `[]`。scalar は collector 概念であって構造の特別化で
  ないことを実測。

## Task 2: 失敗時アクションの候補定義実測 (DR-048) — phase17, +6 test

DR-048: early-exit は持たず完走。完全経路 0 本の失敗時、候補経路で失敗時アクション持ち要素 (help) が
selected なら保持 Error に代えて発火。複数は argv 位置最小の先勝ち。射程外 = 「候補経路で selected」の
精密定義 (どの partial 経路を数えるか)。

### 実装

- `Node::FailMark(name)` (0-token、argv 位置を記録)。fail_action フラグの long satellite を
  `Seq([Exact(trigger), Bind(name,true), FailMark(name)])` に lowering。
- `Branch::Held(msg, depth)` -> **`Held(msg, depth, Array[Binding])`** に拡張 = 失敗時部分状態を保持
  (**F-039 との接続**)。number positional の parse 失敗など scope acc が生きている Held 生成点で acc を
  載せる (help marker が dead end でも捕捉できる)。matcher / leaf の Held は `[]`。
- `Outcome::FireAction(name)` 追加。`parse` に `faildef~ : FailDef` を追加。action marker は
  `@act:<name>` キー (renderer / build_result は skip)。argv 位置は marker の value (VNum)。
- **FailDef 3 案**:
  - `AcceptsOnly` … 候補 = scope-end Accept のみ (dead end 無視)
  - `WithHeld` … + Held 到達経路の partial state (F-039)
  - `DeepestOnly` … 最大 argv 消費の候補経路のみ

### 観測マトリクス (phase17)

| 失敗形 | AcceptsOnly | WithHeld | DeepestOnly | 主眼 |
|---|---|---|---|---|
| `--help --typo` (help 先行) | action:help | action:help | action:help | 到達可能な partial Accept に help -> 全案発火 |
| `--typo --help` (失敗先行) | fail | fail | fail | help 到達不能 (positional 無で pos0 詰み) -> argv 順の効果 |
| `-- --help x` (dd 内部) | fail | fail | fail | 先食い/内部一体で --help は raw 消費、help 未発火 |
| `--help x y` (dd 前) | — | action:help | — | 対照: 先食いが効く位置なら発火 |
| `--help --version --typo` | — | action:help | — | 複数アクション -> argv 先勝ち (help@1 < version@2) |
| `--version --help --typo` | — | action:version | — | 同上、順序反転 |
| **`--help abc`** (num positional 失敗) | **fail (保持 Error)** | **action:help** | **action:help** | **候補定義の分岐点** |
| `abc --help` (失敗先行) | fail | fail | fail | abc が先に数値失敗、help 未到達 |

### 観測から得た知見 (issue への材料)

1. **主要な分岐は AcceptsOnly vs (WithHeld / DeepestOnly)**。`--help abc` で help は number positional の
   Held に dead end する。AcceptsOnly は scope-end Accept のみ数えるので help を見落とし保持 Error を出す。
   WithHeld / DeepestOnly は Held の partial state (F-039) から help を拾い発火する。
2. **WithHeld と DeepestOnly は実質一致する** (E で b==c、F で固定)。理由: 先食いが `--help` の raw 消費を
   禁じるので、`--help` を消費した経路は**必ず** help satellite を撃つ。ゆえに「help を撃たずに深く進む
   候補」が存在せず、「最深候補のみ」と「Held 含む全候補」で差が出ない。**-> 精密定義の実質的な選択は
   「dead end (Held の partial state) を候補に含めるか否か」の 1 点に縮約される** (WithHeld を採るか
   AcceptsOnly を採るか)。DR-048 §2 の狙い (`--help --typo` で古典 exit 相当) は AcceptsOnly でも満たせるが、
   `--help abc` のような「help 後に別要素が即エラー」で古典 CLI が help を出す挙動を再現するには WithHeld が要る。
3. **argv 先勝ちは効果列 (DR-045) の argv 順で素直に実装できた** (marker に位置を載せ min を取るだけ)。
4. **dd 抑制は新規則ゼロで導出** (help satellite が severed 面で発火しない = 既存の 先食い/内部一体則)。

### 契約割れ / flag (メインへ)

## タスク 1: ambiguous 非発火の輪郭固定 (DR-048 §2) — phase18, +1 test

DR-048 確定文言「発火条件は完全経路 0 本のみ、2+ 本 (ambiguous) では発火しない」。第 6 弾の `parse` は
FireAction 判定を `full.length()==0` アームにのみ置いていた (ambiguous = `n` アームは Ambiguous を返す)
ので、**確定仕様を既に満たしていた**。輪郭をテストで固定:

- help (fail_action) + 同一トリガ `--v` が別実体 a/b を束縛する 2 エントリ。`--help --v` は help 発火後
  --v で 2 完全経路 ({help,a}/{help,b}) に分岐 = **ambiguous:2**。help は両経路で selected だが発火しない。

- 判断本質をコメントに明記: ambiguous の本体は競合解釈の提示、0 本失敗 (使い方不明→help 有益) と
  ambiguous (どの解釈か明確化したい→エラーが本体) では見せたい情報が違う (DR-048)。
- 誘導行 (0 本 + ambiguous 両方の末尾 1 行) はレンダリング層で PoC 射程外 (テストせず)。

# 第 8 弾: 3 段以上の中間 shadowing の name-vs-trigger 微差 (DR-042 検証マトリクス最終独立項目)

DR-042 global installer の shadowing 規則を 3 段木で実測し、**DR-042 文言が読み A / B の両方を許す曖昧さ**を
検出した。実測に加え文言精密化の材料提供が目的。コード変更なし (観測のみ、phase19 +3 test)。第 7 弾 98
test → **101/101 pass**。

## 設定

root(global `verbose`, トリガ `--verbose`) → mid(中間宣言) → sub(無宣言) の 3 段 command 木。mid の宣言を
3 通りに変え、`mid sub --verbose`(孫スコープ発火)と `mid --verbose sub`(中間スコープ発火)で束縛先を観測。

## 観測マトリクス (phase19)

| ケース | mid の宣言 | `mid sub --verbose` | `mid --verbose sub` | 意味 |
|---|---|---|---|---|
| 1 | 別 name・同 literal (chatty=--verbose, 非global) | `{mid:{chatty:true,sub:{}}}` | 同左 | root コピーは sub に届かず、mid の chatty が拾う = **読み B** |
| 2 | 同 name・同 literal (verbose=--verbose, 非global) | `{mid:{verbose:true,sub:{}}}` | 同左 | 通常 shadow の 3 段版、mid の verbose が拾う |
| 3 | 同 name・別 literal (verbose=--chatty, 非global) | `{verbose:true,mid:{sub:{}}}` | 同左 | トリガ非衝突 → root コピーが sub まで伝播 (Rooted link → root キー) |

補足: ケース 3 の mid own トリガ `mid --chatty sub` → `{mid:{verbose:true,sub:{}}}` (mid の別セル)。

## 判明した事実

1. **PoC は読み B (伝播断ち) を実装**。root コピーは is_global を保って level-by-level に伝播するが、mid で
   トリガ衝突により止まったコピーは mid の decl に landing しないため、再伝播元にならない → sub に届かない。
   孫の `--verbose` は sub が verbose 入口を持たないため sub 閉じ→mid 再開 (早閉じ抑制) で mid の chatty に
   読まれる (最内 reader が勝つ)。ケース 1 が読み A なら sub は root コピー入口を持ち root verbose セルへ
   link したはずで、結果は `{verbose:true,mid:{sub:{}}}` になる — 実測は `{mid:{chatty:true,sub:{}}}` なので B。
2. **shadow はトリガ literal 判定 (名前非依存)**。ケース 1 (別 name) でも shadow が成立 (mid が止める)、
   ケース 3 (同 name・別 trigger) では shadow せず伝播 = DR-042 の「判定は要素名ではなくトリガ literal」を
   3 段で追認。名前が重なるだけでは shadow しない。

## 契約割れ / flag (最優先) — DR-042 文言が読み A / B の両方を許す

DR-042 global installer の記述: 「子孫 command スコープへ ref/link 衛星の宣言的コピーを追加する。同一トリガ
literal を自前で持つスコープへはコピーしない (shadowing、最小スコープ優先 = lexical 解決のパース時適用)」。

この 1 文の中に**互いに逆を示唆する 2 つの語**が同居している:

- **「子孫スコープへコピー / 自前で持つスコープには入れない」= per-scope フィルタ**として読むと **読み A**:
  sub は `--verbose` を自前宣言していないので、root コピーを受け取るべき (中間 mid の shadow は mid 自身の
  可否にしか効かない)。
- **「最小スコープ優先 = lexical 解決」= subtree 被覆**として読むと **読み B**: lexical scoping では最内宣言が
  配下全体を shadow するので、mid の `--verbose` 宣言が mid の subtree (sub 含む) 全域で `--verbose` を覆い、
  sub の `--verbose` は mid の chatty に解決される。

**PoC は読み B を実装** (lexical 解決と整合、level-by-level 伝播の帰結)。読み B が意味論的に妥当と考えられる
(DR-032/033 の lexical scope と一貫) が、**コピー機構の記述 (per-scope フィルタ) を字義通り読むと読み A に取れる**
ため、文言だけでは 3 段の中間 shadow の挙動が一意に定まらない。

### 精密化の提案 (材料、仕様確定は当事者に委ねる)

DR-042 に「**中間スコープが同一トリガを shadow した場合、その shadow は配下 command 全体に及ぶ (lexical 連鎖) —
孫が自前宣言をしていなくても root のコピーは伝播しない**」を明記すると、per-scope フィルタ読み (A) を排除して
読み B に一意化できる。逆に読み A を意図する場合は「各子孫スコープは独立に、自スコープの宣言のみで可否判定
する (中間の shadow は伝播を断たない)」と明記する必要がある。どちらを canonical にするかは当事者判断。
部外者観測につき、lexical 意味論 (DR-032/033) との整合確認を推奨。
