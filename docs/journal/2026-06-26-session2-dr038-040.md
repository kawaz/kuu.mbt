## 1. ロックした決定 (索引)

1. **パース意味論 = 完全経路の一意性** (DR-038)。最長一致は規則として廃止 (創発に降格)。0/1/2+ 経路 → 失敗/成功/ambiguous。`-n1.0f` は parser が両形式を受理すれば ambiguous。エンジンは **bounded path-search** (2本目の完全解で打ち切り)、greedy-commit ループは置換。
2. **合流テーゼ** (DR-039)。AtomicAST = 既存エンジンのノードグラフのシリアライズ形。三層 = ノードADT(再帰enum) + 探索器 + **name 駆動の結果ビルダー(新層)**。
3. **進め方 = 垂直スライス** (DR-039)。AtomicAST を単独確定せず実装と同時に削り出す。bounded path-search 評価器を最初に作る。JSON Schema は最後。
4. **type registry 運用** (DR-040)。canonical default ← 言語DX ← definitions ロックの3層。方言軸は pre_filters(前処理) / parser(本質字句) に2分。寛容default + regex_match 制限が素の経路。tree-shaking は名前参照+外部注入で自明。

## 3. 次の一手 (推奨)

**環境: ローカル Claude Code を推奨。** 理由: ①リモートはエフェメラルでコンテナ再生時にバックグラウンド処理が死ぬ (本セッションで実際にワークフローが落ちた)。②設計コーパスがリポジトリ外 (scratchpad のみ) で揮発リスク。③`moon build`/`moon test` を回しながら enum を削る作業はローカルの速いループが効く。

**着手順 (DR-039 の垂直スライス):**
1. ノードADT を MoonBit enum で最小定義 (`exact/or/seq/primitive` + `multiple` + `name`)。
2. **bounded path-search 評価器**を実装 (DR-038)。`TryResult{Accept/Reject/Error}`・`make_or_node` を再利用、`parse_raw` の greedy ループは置換。← 最大リスク、最初に検証。
3. name 駆動の結果ビルダー (DR-025「最も浅い name 層」)。
4. 評価器が要求する形に AtomicAST enum を反復 → 固まったら JSON Schema。
