# 2026-07-02 垂直スライス PoC — 評価器コア + installer 連鎖

DR-038/041/042 の契約を「動くコード + 契約を輪郭ごと固定したテスト」で検証する
垂直スライス。製品コードではなく検証装置。場所: `poc/` (MoonBit, `kawaz/slicepoc`)。

toolchain: moon 0.1.20260626 / moonc v0.10.2 nightly。新形式 `moon.mod` / `moon.pkg`
(feature flag `rr_moon_mod,rr_moon_pkg`)。全ソースはルートパッケージ、テストは whitebox
(`*_wbtest.mbt`)。

## アーキテクチャ (三層, DR-039)

1. **ノード ADT** (`node.mbt`): `Exact / StrArg / NumArg / Bind / Or / Seq / Many /
   Scoped(name) / ScopeNode(Scope) / NativeMatch(Matcher) / DdMark`。
   `Scope` は greedy 面 (順不同トリガ) + positional 面 (順序消費) + entities (値セル)。
2. **評価器** (`eval.mbt`): bounded path-search。`eval` が枝 (`Branch = Accept|Held`) を
   全列挙、`parse` が全消費完全経路を数え 0/1/2+ で結末 (DR-038)。
3. **結果**: 現状は flat な `Array[Binding]` (key=value@source)。`Scoped` は path 数に
   透過 (name 駆動ネストは result-builder の軽い課題として据え置き)。

### matcher は「データ」でありクロージャではない (重要な実装判断)

DR-041 の native matcher を**不透明クロージャ**ではなく**データ** (`Matcher` enum:
`EqSplit` / `ShortCombine` + entry 表) として持った。DR-042 の「回収したエントリ表が
matcher の構成データ」と一致し、かつ **AtomicAST 全体が `derive(Eq, Show)` 可能**になる
→ Phase 2 の順序非依存 permutation テストで最終 AST を構造比較できる。クロージャだと
比較不能で permutation 検証が書けなかった。**これは Phase 1 で確定した設計上の勝ち。**

### 先食いの述語 = 「greedy が読むか」(DR-041 §4 の素直な写し)

`greedy_reads(scope, toks, pos)` = 「いずれかの greedy エントリが pos で ≥1 トークン
消費する読みを持つか」。true なら positional の raw 素通し枝を生成しない。DR-041 §4
「次トークンがそのスコープの greedy エントリに読める場合、素通し消費の読みは枝として
生成しない」の逐語的実装。追加のトリガ集合計算は不要だった。

### 完全経路の「重複」= 同一 interpretation として dedup

interleaving 探索は同じ結果に別ルートで到達しうる (例: greedy 発火を Many の 0-rep
前後どちらでも試す)。**完全経路を binding 列の完全一致で dedup** することで false
ambiguous を消す。根拠: DR-038 の「1本」は「1つの解釈」の意味で、binding が同一なら
利用者は区別できない = 1解釈。異なる解釈は異なる binding を生むので dedup は真の
ambiguity を保存する (sort せず完全一致のみ dedup、順序も保存)。

## 契約が割れた/怪しかった箇所

### F-1: dd の `--` マーカーは 先食い の対象外 (DR-041 §4 の文言ギャップ)

DR-041 §4 の 先食い は文言上 **「greedy エントリ」に読めるトークン**だけを raw 素通しから
守る。ところが dd の `--` は**構造要素であって greedy ではない** (DR-041: 「dd は matcher を
持たない」)。

厳密実装すると: dd が optional で継続 positional が寛容 (`Many(StrArg)` 等) の場合、
リテラル `--` が出現すると

- **dd-present 枝**: `--` を消費 → 継続を greedy 断ち子スコープで消費
- **dd-absent 枝**: `--` を positional が raw 消費 (先食いは greedy しか見ないので抑制されない)

の**両方が全消費完成 → ambiguous** になる。つまり `prog -a -- -b` すら曖昧になり、
「`prog -a -- -b` と `prog -- -a -b` で解釈が変わる」という DR-042 dd の狙いが**壊れる**。

**PoC の解決 (2 段階で確定)**:

1. **失敗した第 1 案** (位置局所の特権化): DdMark を Or 的に present/absent 枝で持ち、
   「現トークンが `--` なら present 枝のみ」とした。**これは漏れる**。`prog -a -- -b`
   では -a 発火後に DdMark へ到達する経路 (present, 正) と、DdMark absent 枝が継続を
   親 greedy 込みで inline してから深い位置で `--` を raw 消費する経路が**両立**し、
   Phase 2 の dd テストが `ambiguous:2` になった (`{a,x:-b}` と `{a,b,x:--}`)。
   位置局所の特権化では、absent 枝が `--` をスキップした先で `--` が食われるのを
   止められない。
2. **確定案** (severed モード + `--` を抑制トリガに): scope 走査に `severed : Bool` を
   通す。`--` を消費したら以降 severed=true (greedy 面 off、全て raw)。かつ 先食い
   述語 `suppressed` を「greedy が読む **または** (`token=="--"` かつ scope が dd を
   持ち未 sever)」に拡張。これで `--` はどの深さでも raw 消費されず、absent 枝の漏れが
   消える。DdMark は payload なしの純マーカーに単純化 (継続は後続 positional 要素そのもの)。

結果、解釈変化が素直に通る: `-a -- -b` → `a=true, x=-b` / `-a -b c` → `a=true, b=true, x=c`
(`--` の有無で -b の解釈が反転)。

**メインへ (DR 化候補)**: DR-041 §4 の 先食い を「greedy エントリに読める」だけでなく
**「スコープの構造トリガ (greedy ∪ 未 sever の dd marker `--`)」**に拡張すべき。現状文言
(greedy 限定) のままだと寛容 positional + dd で曖昧を量産する。「dd は matcher を持たない
純構造」(DR-041) は保ったまま、抑制トリガ集合にだけ `--` を載せる形が最小侵襲。

### 契約が割れた/怪しかった箇所 (Phase 2)

**F-2: dd installer は構造を作らず「語彙の licensing」に縮退した (installer 署名の発見)**。
long/short は具体構造 (exact 衛星 / matcher データ) を build に積む。ところが dd は:
継続 = 後続 positional 要素そのもの、再スコープ = 評価器の severed モード、marker 配置 =
core lowering (`init_build`)。結果 **dd installer の実務は `is_dd` の回収 (= 除去) だけ**に
なった。「installer が再スコープを植え付ける」(DR-042) の実装像は、**installer=語彙所有 /
評価器=意味論 / core=マーカー lowering** の 3 者分担に自然に割れた。これは DR-042 が
「射程外」とした installer シグネチャ・座席の実測データ。long/short (構造構築型) と
dd (語彙 licensing 型) は**別カテゴリの installer** と見なすのが素直。

### matcher の座席 (task 3 の問い) — 実測回答

native matcher は **greedy 面のエントリ (scope.greedy 配列中の 1 Node = `NativeMatch`)**
として落ちた。filter chain の席でも専用席でもなく、exact 衛星と**同列の greedy 市民**。
根拠: 先食い述語 `greedy_reads` が「greedy エントリが読むか」で matcher も exact 衛星も
一様に扱えたため、両者を同じ配列に混ぜるのが最も素直だった。

副次発見: short の「単一文字 `-v` / `-p 8080`」も ShortCombine matcher が被覆するため、
**short installer は別途 exact 衛星を積まず matcher 1 個だけ**になった (DR-042 の「greedy
衛星 + matcher」のうち衛星が matcher に吸収)。long は `--k=v` (eq-split matcher) と
`--k v` (exact 衛星) が別機構なので**両方**積む。short と long で衛星の要否が非対称。

### installer シグネチャ (自然に落ちた形)

`(Definition, Build) -> Unit`。Definition から所有語彙を回収 (mutable 属性を空に) し、
Build (greedy: 追加専用 / entities: env_key を in-place 宣言 / positional: core 所有) に
寄与。値源系 (env) は entity の席 (`env_key`) を宣言するのみ (ラダー順は評価器所有、
不変則④)。順序非依存は「greedy は追加専用・entities は名前で in-place・positional は
dd 以外触らない」から出た。中央 gather-then-build は不要だった (DR-042 の判断を追認)。

## Task 1: dd を greedy exact 衛星に再 encode（先食い特例の撤回）

第 1 弾では F-1 として「先食いを (greedy ∪ dd マーカー) に拡張すべき」と報告したが、
**DR-041 改稿はこの特例を却下**し、dd を **greedy 面の住人 (トリガ兼消費者)** と捉え直した。
`--` が値に食われないのは「dd が greedy エントリだから 先食いが効く」、以降の option 抑制は
「発火した greedy の内部消費は一体」から導出 — **dd 専用規則ゼロ**。

実装追従:
- `DdMark`（positional 純マーカー）→ `DdSat(String)`（greedy 面の exact 衛星）。
- `suppressed` を **greedy-only** に戻す（`scope_has_dd` / `token=="--"` 特例を削除）。
  `--` の抑制は「DdSat が greedy エントリとして `--` を読む」→ `greedy_reads` が true から出る。
- greedy 発火ステップで **DdSat が発火したら severed=true で継続**（内部一体則の実装）。
  `severed` フラグは DR-041 が明示的に許容する encoding（「経路局所の sever フラグ + 純マーカー」）。
- core lowering: dd positional は**背骨に載せない**（inst_dd が DdSat を greedy に追加）。

**確認**: 既存 dd テスト（`-a -- -b`→`a,x=-b` / `-a -b c`→`a,b,x=c`）が**特例なしで通る**
（21/21 維持）。第 1 弾 F-1 の「特例が要る」は spec 改稿で解消 — dd=greedy 住人の捉え直しが
正しく、第 1 弾の特例は不要だったことを実測で確認。add-if-absent (`add_greedy`) も先行導入。
